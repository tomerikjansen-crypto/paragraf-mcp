# ADR-003: Sitatvalidering, endringshistorikk og utvidede rettskilder

**Status:** Utkast
**Dato:** 2026-03-09
**Beslutningstagere:** Utviklingsteam
**Kontekst:** Utvidelse av paragraf med sitatvalidering, gyldighetssjekk, EU/EØS-referanser og forarbeider

---

## Sammendrag

Dette dokumentet beskriver arkitektur og designbeslutninger for fire relaterte utvidelser:

1. **Sitatvalidering (`validate_citation`)** — verifisere at en paragraf-referanse faktisk eksisterer
2. **Gyldighets- og endringshistorikk (`check_currency`)** — eksplisitt eksponere om en paragraf er gjeldende, er blitt endret, eller vil bli endret av en vedtatt endringslov
3. **EU/EØS-kryssreferanser** — indeksere EU-direktiver og -forordninger som siteres i norsk lovtekst
4. **Forarbeider og offentlige utredninger** — NOU, Prop., Innst. fra eksterne API-er

Ingen av disse er implementert. Dokumentet er beslutningsgrunnlag for fremtidig implementasjon.

## ADR-003.1: Sitatvalidering

### Kontekst

MCP-verktøyene `lov()`, `forskrift()` og `hent_flere()` er i dag blinde for om en forespurt paragraf eksisterer. En LLM som siterer feil paragrafnummer (f.eks. `§ 14-99` i stedet for `§ 14-9`) får tom respons uten forklarende kontekst.

Ingen `parse_citation()` eller `validate_citation()`-funksjon finnes. Det nærmeste er `_format_based_on()` i `service.py`, som parser `based_on`-feltet for hjemmelslover — men dette er ikke generell sitatvalidering.

### Hva sitatvalidering gir

```
Input:  "aml § 14-9"
Output: { dok_id: "lov/2005-06-17-62", section_id: "14-9", exists: true, canonical: "§ 14-9" }

Input:  "aml § 14-99"
Output: { exists: false, suggestion: ["§ 14-9", "§ 14-10"] }
```

Fordelen over dagens system:
- **Tidlig feiling med kontekst** — LLM kan korrigere seg selv i én runde fremfor å sende tom respons
- **Kanonisk form** — normaliserer variasjoner ("§14-9", "§ 14-9", "paragraf 14-9")
- **Batch-validering** — spesielt nyttig for `hent_flere()` med mange referanser

### Teknisk tilnærming

Krever lite ny kode:
1. **Regex-mønster** for å parse sitatstrenger til `(dok_id, section_id)`-par
2. **Ett DB-oppslag** mot `(dok_id, section_id)` i `lovdata_sections`
3. Intern hjelpefunksjon i `service.py` — eller eksponert som MCP-verktøy `valider_sitat(id, paragraf)`

Støtteformater:
- `aml § 14-9` (alias + paragraf)
- `lov/2005-06-17-62/§14-9` (direkte dok_id)
- `§ 14-9` (paragraf alene, forutsatt kjent lov)
- `Artikkel 5`, `Kapittel 3` (ikke-numeriske section_id-er)

### Prioritet

Lav–medium. Nyttig for robusthet, enkelt å implementere.

## ADR-003.2: Gyldighets- og endringshistorikk (`check_currency`)

### Kontekst

I dag har `lovdata_documents` et `is_current`-felt (boolean) som angir om et *dokument* er opphevet. Dette er grovkornet og dekker ikke:

- Hvilken endringslov som sist endret en *paragraf*, og når
- Om en paragraf er formelt gjeldende i dag, men vil bli endret av en allerede vedtatt endringslov med fremtidig ikrafttredelse
- Dato for ikrafttredelse av vedtatte, men ikke iverksatte endringer

Dette er kritisk juridisk informasjon. En LLM som svarer basert på gjeldende lovtekst uten å opplyse om at paragrafen er endret fra neste år, gir et misvisende svar.

### De tre tilfellene

```
1. Paragraf er gjeldende og uendret nylig
   → ingen merknad

2. Paragraf ble endret [dato] ved [endringslov]
   → "Merk: § 14-9 ble endret 2024-01-01 (lov 2023-06-16-50)"

3. Paragraf er gjeldende i dag, men endres [fremtidig dato]
   → "Merk: § 14-9 endres fra 2026-07-01 (lov 2025-12-19-120, ikke iverksatt)"
   → evt. "Ikrafttredelse fastsettes av Kongen" hvis dato ikke er satt
```

Tilfelle 3 er spesielt viktig: en vedtatt endringslov kan ligge i databasen med `is_amendment=TRUE` og `is_current=TRUE`, uten at det fremgår at den berører en spesifikk paragraf i en annen lov.

### Hva som mangler i dagens datamodell

| Felt | Finnes | Mangler |
|---|---|---|
| `is_current` (dokument-nivå) | Ja (`lovdata_documents`) | — |
| `is_amendment` (dokument-nivå) | Ja | — |
| Hvilken paragraf endringsloven endrer | Nei | Kobling `endringslov → (dok_id, section_id)` |
| Dato for ikrafttredelse av endring | Nei | Per-seksjon ikrafttredelse |
| Historisk versjon av paragraf | Nei | Versjonstabell |

### Hva Lovdata-XML sannsynligvis inneholder

**Ikke verifisert** — må sjekkes i XML-strukturen og `structure_parser.py`. Lovdata-dokumenter inneholder trolig:
- `changedBy` eller tilsvarende metadata på seksjonsnivå
- `effectiveDate` per endring
- Referanse tilbake til endringsdokumentets `dok_id`

```sql
-- Verifiser: finnes endringsreferanser i sections-tabellen?
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_name = 'lovdata_sections';

-- Finnes ikrafttredelsesdato-felt noe sted?
SELECT column_name FROM information_schema.columns
WHERE table_name IN ('lovdata_documents', 'lovdata_sections', 'lovdata_structure')
  AND column_name ~* 'date|dato|ikraft|effect|change|amend';
```

### Foreslått databaseskjema

```sql
-- Ny tabell: endringer på paragrafnivå
CREATE TABLE section_amendments (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dok_id            TEXT NOT NULL,                          -- loven som endres
  section_id        TEXT NOT NULL,                          -- paragrafen som endres
  amending_dok_id   TEXT NOT NULL REFERENCES lovdata_documents(dok_id),
                                                            -- endringsloven
  effective_date    DATE,                                   -- NULL = ikke fastsatt
  effective_note    TEXT,                                   -- "fastsettes av Kongen" e.l.
  is_future         BOOLEAN GENERATED ALWAYS AS            -- praktisk flagg
                    (effective_date IS NULL OR effective_date > CURRENT_DATE) STORED,
  FOREIGN KEY (dok_id, section_id) REFERENCES lovdata_sections(dok_id, section_id)
    ON DELETE CASCADE
);

CREATE INDEX section_amendments_dok_section_idx
  ON section_amendments(dok_id, section_id);
CREATE INDEX section_amendments_future_idx
  ON section_amendments(is_future) WHERE is_future = TRUE;
```

### Nytt MCP-verktøy

```
check_currency(lov_id, paragraf?)
  → Sjekk om angitt paragraf (eller lov som helhet) er gjeldende,
    nylig endret, eller vil endres av vedtatt endringslov.

  Returnerer:
  - is_current: bool
  - last_amended: { date, amending_law_id, amending_law_title } | null
  - pending_changes: [{ amending_law_id, amending_law_title,
                        effective_date | null, effective_note }]
```

Alternativt integreres informasjonen i `lov()`-responsen som en `endringsinfo`-blokk, slik at LLM-en alltid ser det uten eksplisitt kall.

### Åpne spørsmål

1. **XML-struktur:** Finnes `changedBy`/`effectiveDate` på seksjonsnivå i Lovdata-XML? (`structure_parser.py` og XML-dokumentasjonen i `../unified-timeline/tredjepart-api/lovdata-xml.md` bør sjekkes)
2. **Granularitet:** Er endringer alltid knyttet til en spesifikk paragraf, eller kan en endringslov endre et kapittel uten å liste enkeltparagrafer?
3. **Historiske versjoner:** Ønskelig å lagre tidligere versjon av paragrafteksten, eller kun flagge at endring har skjedd?
4. **Fremtidig ikrafttredelse:** Lovdata-XML kan mangle ikrafttredelsesdato for endringer som fastsettes ved kongelig resolusjon — håndtering av NULL-dato er viktig.

## ADR-003.3: EU/EØS-kryssreferanser

### Kontekst

Mange norske lover implementerer EU-direktiver og -forordninger via EØS-avtalen. Disse referansene opptrer trolig i lovteksten, f.eks.:

```
"EØS-avtalen vedlegg XVIII nr. 1 (direktiv 2008/104/EF)"
"forordning (EU) 2019/1239 (eFTI)"
"direktiv 2011/36/EU"
```

**OBS — ikke verifisert:** Det er ikke bekreftet at disse referansene opptrer i `content`-feltet i `lovdata_sections`, fremfor kun i titler, merknader eller `based_on`-feltet.

```sql
-- Kjør mot unified-timeline / Supabase før implementasjon
SELECT dok_id, section_id, LEFT(content, 400) AS snippet
FROM lovdata_sections
WHERE content ~* 'direktiv\s+\d{4}/\d+/E[UØF]'
   OR content ~* 'forordning\s+\(EU\)'
   OR content ~* 'EØS-avtalen\s+vedlegg'
LIMIT 20;

SELECT COUNT(*) FROM lovdata_sections
WHERE content ~* 'direktiv\s+\d{4}/\d+/E[UØF]'
   OR content ~* 'forordning\s+\(EU\)';
```

### Alternativer vurdert

| Alternativ | Dekning | Kompleksitet | Avhengigheter |
|---|---|---|---|
| A: Regex over `content` ved sync | ~80–90% | Lav | Ingen |
| B: Regex + manuell kuratering | ~95% | Medium | Redaksjonell innsats |
| C: NLP-basert entitetsgjenkjenning | ~95–99% | Høy | ML-modell |
| D: Ekstern EUR-Lex API ved behov | 100% | Lav | Nettverkskall i MCP |

### Anbefalt beslutning

**Alternativ A** som første steg. CELEX-URI kan beregnes deterministisk — ingen eksternt API-kall nødvendig.

### CELEX-beregning (deterministisk)

| Sitatformat | Type | CELEX | EUR-Lex-URI |
|---|---|---|---|
| `direktiv 2008/104/EF` | L | `32008L0104` | `.../CELEX:32008L0104` |
| `direktiv 2011/36/EU` | L | `32011L0036` | `.../CELEX:32011L0036` |
| `forordning (EU) 2019/1239` | R | `32019R1239` | `.../CELEX:32019R1239` |
| `forordning (EF) 1071/2009` | R | `32009R1071` | `.../CELEX:32009R1071` |

Mønster: `3{år}{type}{nummer_null-paddet_til_4}`  — Direktiv=`L`, Forordning=`R`, Beslutning=`D`

### Foreslått databaseskjema

```sql
CREATE TABLE eu_references (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dok_id       TEXT NOT NULL REFERENCES lovdata_documents(dok_id) ON DELETE CASCADE,
  section_id   TEXT,
  eu_type      TEXT NOT NULL,   -- 'direktiv', 'forordning', 'beslutning'
  eu_number    TEXT NOT NULL,   -- '2008/104/EF'
  eu_year      INT,
  celex        TEXT,            -- '32008L0104'
  eur_lex_uri  TEXT,
  raw_text     TEXT,
  extracted_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX eu_references_dok_id_idx ON eu_references(dok_id);
CREATE INDEX eu_references_celex_idx ON eu_references(celex);
```

### Nytt MCP-verktøy

```
eu_referanser(lov_id, paragraf?)
  → List EU/EØS-direktiver og -forordninger som refereres i angitt lov/paragraf.
  → Returnerer: eu_type, eu_number, celex, eur_lex_uri, section_id
```

## ADR-003.4: Forarbeider og offentlige utredninger (NOU/Prop./Innst.)

### Kontekst

Paragraf dekker kun gjeldende lover og sentrale forskrifter (gratis Lovdata API). Forarbeider er et kjent gap. For å forstå lovgivers intensjon er NOU, Proposisjoner og Stortingsinnstillinger sentrale rettskilder.

### API-kilder undersøkt (2026-03-09)

| Kilde | NOU | Prop./Meld.St. | Innst. | API | Auth | Format | Lisens |
|---|---|---|---|---|---|---|---|
| **KUDOS** (kudos.dfo.no) | 777 dok. | 4 853+ | Ukjent | v1, tidlig tilgang | Ukjent | Ukjent | Trolig åpen |
| **Stortinget** (data.stortinget.no) | Nei | Ja (som saker) | Ja | Dokumentert | Ingen | XML/JSON | NLOD |
| **data.regjeringen.no** | Ja | Ja | Nei | **Lagt ned juli 2022** | — | — | — |
| **Nasjonalbiblioteket** (nb.no) | Historiske skanninger | Nei | Nei | Begrenset | Begrenset | Bilde/tekst | Begrenset |
| **data.norge.no** | Katalog | Katalog | Nei | Nei | — | — | — |

### KUDOS (kudos.dfo.no) — mest lovende

DFØ-plattform med 40 000+ kunnskapsdokumenter fra offentlig sektor:

- NOU: 777 dokumenter
- Proposisjon til Stortinget: 4 853
- Stortingsmelding: 2 463
- Evaluering: 6 849
- Riksrevisjonsrapport: 508

API-dokumentasjon: `kudos.dfo.no/docs/api` (v1, tidlig tilgang). Digdir har brukt KUDOS åpne data til KI-eksperimenter, noe som tyder på at bulk-tilgang er mulig.

**Kritisk ukjent:** Om API-et gir fulltekst eller kun metadata.

**Neste steg:** Undersøk `kudos.dfo.no/docs/api` direkte. Kontakt [email protected] for detaljer om fulltekst-tilgang.

### Stortinget API (data.stortinget.no) — produksjonsklar for Innst./Prop.

- Veldokumentert, ingen autentisering, 100 req/min (unntak kan innvilges)
- NLOD-lisens — samme som Lovdata
- Dekning fra ~2008 for XML-dokumenter

Tilgjengelige dokumenttyper:
- `innstilling` (Innst.)
- `proposisjon` og `melding` som saker (`saksbehandling`-data)
- `representantforslag` (Dok. 8)
- `lovvedtak`

**Eksempel-endepunkter:**
```
https://data.stortinget.no/eksport/publikasjoner?publikasjontype=innstilling&sesjonid=2023-2024&format=json
https://data.stortinget.no/eksport/saker?sesjonid=2023-2024&format=json
```

**Begrensning:** NOU finnes ikke i Stortingets API — NOU produseres av regjeringen, ikke Stortinget. Returnerer XML ("trykkeriformat"), ikke strukturert JSON — krever XML-parser tilsvarende `structure_parser.py`.

### Alternativ: regjeringen.no web-scraping

regjeringen.no publiserer NOU og Prop. som HTML per kapittel og PDF. Ingen API (data.regjeringen.no ble lagt ned juli 2022). Web-scraping er mulig, men ustabilt og ikke offisielt støttet. Ikke anbefalt som primærkilde.

### Arkitektonisk vurdering for paragraf

Forarbeider er fundamentalt annerledes enn gjeldende rett:
- **Omfang:** Kan være svært lange (NOU på 500+ sider)
- **Bruksmønster:** Sjelden trengt i sin helhet — vanligvis relevante avsnitt til en bestemt paragraf
- **Kobling:** Verdien ligger i koblingen `forarbeid ↔ lovparagraf`, ikke i fulltekst-søk isolert

**Anbefalte neste steg:**
1. Avklar KUDOS fulltekst-tilgang
2. Test Stortinget API for Innst. + Prop. (produksjonsklar)
3. Vurder om forarbeider skal lagres i paragraf-databasen eller hentes on-demand via MCP-kall til dedikert tjeneste
