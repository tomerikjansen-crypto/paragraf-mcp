# ADR-003: Sitatvalidering og EU/EØS-kryssreferanser

**Status:** Utkast
**Dato:** 2026-03-09
**Beslutningstagere:** Utviklingsteam
**Kontekst:** Utvidelse av paragraf med sitatvalidering og støtte for EU/EØS-referanser

---

## Sammendrag

Dette dokumentet beskriver arkitektur og designbeslutninger for to relaterte utvidelser:

1. **Sitatvalidering (`validate_citation`)** — verifisere at en paragraf-referanse faktisk eksisterer i databasen
2. **EU/EØS-kryssreferanser** — indeksere og eksponere referanser til EU-direktiver og -forordninger som opptrer i norsk lovtekst

Begge funksjonene er foreløpig **ikke implementert**. Dette dokumentet er et beslutningsgrunnlag for fremtidig implementasjon.

---

## ADR-003.1: Sitatvalidering

### Kontekst

I dag er MCP-verktøyene `lov()`, `forskrift()` og `hent_flere()` blinde for om en forespurt paragraf eksisterer. En LLM som siterer feil paragrafnummer (f.eks. `§ 14-99` i stedet for `§ 14-9`) får en tom eller feilrespons uten forklarende kontekst.

Ingen `parse_citation()` eller `validate_citation()`-funksjon finnes. Det nærmeste er `_format_based_on()` i `service.py`, som parser `based_on`-feltet (hjemmelslover i forskrifter) — men denne håndterer ikke sitatvalidering generelt.

### Hva sitatvalidering gir

```
Input:  "aml § 14-9"
Output: { dok_id: "lov/2005-06-17-62", section_id: "14-9", exists: true, canonical: "§ 14-9" }

Input:  "aml § 14-99"
Output: { exists: false, suggestion: ["§ 14-9", "§ 14-10"] }
```

Fordelen over dagens system:
- **Tidlig feiling med kontekst** — LLM kan korrigere seg selv i én runde
- **Kanonisk form** — normaliserer variasjoner ("§14-9", "§ 14-9", "paragraf 14-9")
- **Batch-validering** — spesielt nyttig for `hent_flere()` med mange referanser

### Teknisk tilnærming

Krever svært lite ny kode:
1. **Regex-mønster** for å parse sitatstrenger til `(dok_id, section_id)`-par
2. **Ett DB-oppslag** mot `(dok_id, section_id)` i `lovdata_sections`
3. Evt. MCP-verktøy `valider_sitat(id, paragraf)` eller intern bruk i `lov()`/`hent_flere()`

Støtteformater som bør gjenkjennes:
- `aml § 14-9` (alias + paragraf)
- `lov/2005-06-17-62/§14-9` (direkte dok_id)
- `§ 14-9` (paragraf alene, gitt kjent lov)
- `Artikkel 5`, `Kapittel 3` (ikke-numeriske section_id-er)

### Prioritet

Lav–medium. Nyttig for robusthet, men ikke kritisk. Kan implementeres som intern hjelpefunksjon uten nytt MCP-verktøy.

## ADR-003.2: EU/EØS-kryssreferanser

### Kontekst

Mange norske lover implementerer EU-direktiver og -forordninger via EØS-avtalen. Disse referansene opptrer i lovteksten, f.eks.:

```
"EØS-avtalen vedlegg XVIII nr. 1 (direktiv 2008/104/EF)"
"forordning (EU) 2019/1239 (eFTI)"
"direktiv 2011/36/EU"
```

**OBS — ikke verifisert:** Det er ikke bekreftet at disse referansene faktisk opptrer i `content`-feltet i `lovdata_sections`, fremfor kun i titler, merknader eller `based_on`-feltet. Dette må sjekkes med direkte SQL-spørringer mot databasen før implementasjon.

```sql
-- Verifiseringsquery (kjør mot unified-timeline / Supabase)
SELECT dok_id, section_id, LEFT(content, 400) AS snippet
FROM lovdata_sections
WHERE content ~* 'direktiv\s+\d{4}/\d+/E[UØF]'
   OR content ~* 'forordning\s+\(EU\)'
   OR content ~* 'EØS-avtalen\s+vedlegg'
LIMIT 20;
```

### Alternativer vurdert

| Alternativ | Dekning | Kompleksitet | Avhengigheter |
|---|---|---|---|
| A: Regex over `content` ved sync | ~80–90% av maskinlesbare ref. | Lav | Ingen |
| B: Regex + manuell kuratering | ~95% | Medium | Redaksjonell innsats |
| C: NLP-basert entitetsgjenkjenning | ~95–99% | Høy | ML-modell |
| D: Ekstern EUR-Lex API ved behov | 100% (for eksisterende dir.) | Lav | Nettverkskall i MCP |

### Anbefalt beslutning

**Alternativ A** (regex ved sync) som første steg, med mulighet for å supplere med **B** (manuell kuratering) for kjente direktiver.

**Begrunnelse:**
- Regex dekker de strukturerte referanseformatene som er mest vanlige
- Ingen nye avhengigheter
- CELEX-URI kan beregnes deterministisk — ingen eksternt API-kall nødvendig (se under)
- Alternativ D (live EUR-Lex) unngås som MCP-verktøy fordi det introduserer nettverkslatens og ekstern avhengighet

### CELEX-beregning (deterministisk, ingen API nødvendig)

EU-direktiver og -forordninger har et CELEX-nummer som kan beregnes fra sitatstrengen:

| Sitatformat | Type | CELEX-nummer | EUR-Lex-URI |
|---|---|---|---|
| `direktiv 2008/104/EF` | L (legislation) | `32008L0104` | `.../CELEX:32008L0104` |
| `direktiv 2011/36/EU` | L | `32011L0036` | `.../CELEX:32011L0036` |
| `forordning (EU) 2019/1239` | R (regulation) | `32019R1239` | `.../CELEX:32019R1239` |
| `forordning (EF) 1071/2009` | R | `32009R1071` | `.../CELEX:32009R1071` |

Mønster: `3{år}{type}{nummer_null-paddet_til_4_siffer}`
- Direktiv → `L`
- Forordning → `R`
- Beslutning → `D`

Eksempel: `direktiv 2008/104/EF` → år=2008, type=L, nummer=0104 → `32008L0104`

### Foreslått databaseskjema

```sql
CREATE TABLE eu_references (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  dok_id       TEXT NOT NULL REFERENCES lovdata_documents(dok_id) ON DELETE CASCADE,
  section_id   TEXT,                        -- NULL = dokument-nivå-referanse
  eu_type      TEXT NOT NULL,               -- 'direktiv', 'forordning', 'beslutning'
  eu_number    TEXT NOT NULL,               -- '2008/104/EF'
  eu_year      INT,
  celex        TEXT,                        -- '32008L0104' (beregnet)
  eur_lex_uri  TEXT,                        -- full URI til EUR-Lex
  raw_text     TEXT,                        -- original sitatstreng fra lovtekst
  extracted_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX eu_references_dok_id_idx ON eu_references(dok_id);
CREATE INDEX eu_references_celex_idx ON eu_references(celex);
```

### Nytt MCP-verktøy (forslag)

```
eu_referanser(lov_id, paragraf?)
  → List alle EU/EØS-direktiver og -forordninger som refereres
    i angitt lov, evt. begrenset til én paragraf.
  → Returnerer: eu_type, eu_number, celex, eur_lex_uri, section_id
```

Alternativt kan informasjonen inkluderes i eksisterende `lov()`/`forskrift()`-respons uten nytt verktøy.

### Åpne spørsmål (må avklares før implementasjon)

1. **Verifiser innhold:** Opptrer EU-referanser i `content`-feltet, eller kun i titler/merknader? (SQL-spørring over)
2. **Omfang:** Hvor mange paragrafer inneholder EU-referanser? Estimat trenger en `COUNT`-spørring.
3. **Fullstendighet:** Dekker regex de faktiske referanseformatene i Lovdata-XML, eller brukes andre formater?
4. **Norsk fulltekst:** EUR-Lex har norsk EØS-tekst for mange dokumenter. Er det ønskelig å hente og lagre fulltekst lokalt (stor datamengde)?
5. **Oppdatering:** EU-referanser er stabile (direktiver endres ikke i nummer), men nye lover vil ha nye referanser — dette håndteres naturlig av sync-prosessen.

## ADR-003.3: Forarbeider og offentlige utredninger (NOU/Prop./Innst.)

### Kontekst (fra research 2026-03-09)

Paragraf dekker i dag kun gjeldende lover og sentrale forskrifter (gratis Lovdata API). Forarbeider — NOU, Proposisjoner, Stortingsinnstillinger — er ikke tilgjengelig. Dette er et kjent og bevisst gap (CLAUDE.md).

Følgende API-kilder ble undersøkt:

| Kilde | NOU | Prop./Meld.St. | Innst. | API | Status |
|---|---|---|---|---|---|
| **KUDOS (kudos.dfo.no)** | Ja (777) | Ja (4 853+) | Ukjent | v1, tidlig tilgang | Mest lovende — kontakt [email protected] |
| **Stortinget (data.stortinget.no)** | Nei | Ja (som saker) | Ja | Veldokumentert, NLOD | Produksjonsklar |
| **data.regjeringen.no** | Ja | Ja | Nei | Lagt ned juli 2022 | Ikke tilgjengelig |
| **Nasjonalbiblioteket** | Historiske | Nei | Nei | Begrenset tilgang | Ikke egnet |

### Anbefalte neste steg

1. **KUDOS API:** Undersøk om API-et gir fulltekst (ikke bare metadata). Kontakt [email protected]. Hvis ja, dekker det NOU + Prop. + Meld. St. i én integrasjon.
2. **Stortinget API:** Veldokumentert, ingen auth, 100 req/min, NLOD-lisens. Egnet for Innst. + Prop. som XML.

### Begrensning

Stortinget API returnerer publikasjoner som XML ("trykkeriformat") — ikke strukturert JSON. Parsing vil kreve dedikert XML-parser tilsvarende `structure_parser.py`.
